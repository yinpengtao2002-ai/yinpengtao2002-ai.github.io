# 全站优化问题清单

> 用途：这份文档不是最终设计稿，而是后续逐条评估网站优化方向的工作清单。每一项都先作为“问题”保留，后续再判断是保留、放弃、延后，还是先做 Demo。

## 使用方式

- 状态可选：`待评估` / `保留` / `放弃` / `延后` / `需要 Demo` / `已完成`
- 每次只推进少量问题，避免一次性重构导致已有模型体验被打乱。
- 财务模型本体目前优先保持稳定，除非是明确 bug 或移动端体验问题。

## 总体判断

当前网站已经有比较完整的作品和财务模型，但下一阶段最值得解决的不是继续堆组件，而是让访问者更快理解主站的能力结构：

> Lucas 如何从业务问题出发，把经营分析、财务模型、分析判断和可视化表达连接成一条清晰的能力链路。

AI 工作流和思考方法不强行塞进这条主链路里，它们更适合作为独立板块展示：AI 工作流证明工具使用和效率放大能力，思考方法证明判断方式和持续沉淀。

招聘方、老板、同事或猎头不一定会长时间研究每个模型，所以首页和入口页必须在很短时间内传达“能力路径”，而不是只展示“我做了很多东西”。

## P0：最值得优先评估

### 1. 首页是否需要更清晰的能力链路？

- 状态：`保留（收窄范围）`
- 候选结论：`保留 / 放弃 / 延后 / 需要 Demo`
- 问题：当前首页是否已经足够清楚地表达“业务问题 -> 财务模型 -> 分析判断 -> 可视化表达”？
- 背景：现在各个模块都已经有一定完成度，但主链路不应该无限延伸。AI 工作流和思考方法是独立界面，可以在首页出现入口，但不需要被硬塞进财务能力链路。
- 阶段性结论：
  - 首页主线保留为：`业务问题 -> 财务模型 -> 分析判断 -> 可视化表达`。
  - AI 工作流单独拆成板块，不作为财务链路的最后一环。
  - 思考方法单独拆成板块，不作为财务链路的最后一环。
  - 首页可以展示三个入口：财务模型、AI 工作流、思考与方法，但它们的叙事关系要清楚，不要互相抢主线。
- 可能动作：
  - 重排首页信息顺序，让每一屏只回答一个核心问题。
  - 减少重复按钮和重复说明，把财务主线集中到“业务问题、模型、判断、可视化”。
  - AI 工作流和思考方法只做独立入口与简短预告，不参与财务主链路。
- 验证方式：
  - 让首次访问者在 30 秒内说出这个网站最想证明 Lucas 的什么能力。
  - 检查桌面端和手机端是否都能快速看懂主线。
- 备注：
  - 后续评估时不要再把“AI 工作流”和“思考方法”强行合进第 1 条，它们分别进入第 3 条和第 4 条讨论。

#### 第 1 条拆分列表

##### 首页财务能力主链路

1. 业务问题
   - 要回答：我到底在解决什么经营问题？
   - 例子：预算为什么偏了、利润为什么变了、趋势哪里异常、哪个变量最敏感。

2. 财务模型
   - 要回答：我用什么工具把问题结构化？
   - 例子：预算实际对比、单车指标变动归因、分月趋势分析、利润敏感性分析。

3. 分析判断
   - 要回答：模型结果说明了什么，背后的经营原因是什么？
   - 例子：结构影响、单车边际变化、销量规模变化、固定扣减影响、同比和环比异常。

4. 可视化表达
   - 要回答：我如何让别人快速看懂这个判断？
   - 例子：瀑布图、趋势图、热力图、结构占比、预算对比图、敏感性图表。

##### AI 工作流独立切块

1. 模型使用助手
   - 说明每个财务模型怎么上传、怎么看、适合问什么。

2. 分析提示词
   - 沉淀用于预算复盘、趋势解释、异常归因、管理层汇报的提示词。

3. 效率工作流
   - 展示如何用 AI 辅助整理资料、生成分析初稿、构建汇报框架。

4. 模型选择与边界
   - 记录不同 AI 模型适合做什么、不适合做什么，以及如何校验输出。

##### 思考方法独立切块

1. 业务问题拆解
   - 记录如何从模糊问题拆成可分析、可建模、可验证的问题。

2. 财务建模方法
   - 记录模型结构、指标口径、数据清洗、归因逻辑等方法。

3. 分析判断方法
   - 记录如何从数字变化推导业务判断，避免只看表面涨跌。

4. 可视化表达方法
   - 记录如何选择图表、安排信息层级，让结论更容易被理解。

5. 职场新人观察
   - 记录作为新人在业务、协作、工具学习中的观察，但表达要谦虚、克制。

### 2. Hero 首屏是否应该加入“业务问题 -> 财务模型”的吸引入口？

- 状态：`保留（仅优化 Hero 首屏，第二屏财务模型区暂不改变）`

- 背景：这里不是修改首页第二屏的财务模型区。第二屏仍然暂时保留现在的模型名称、模型预览和列表展示。真正需要优化的是首屏 Hero：它承担第一印象，需要更快把“业务问题”和“模型能力”连起来。
- 阶段性结论：
  - Hero 首屏可以加入：`业务问题 -> 使用某个财务模型`。
  - 首页第二屏财务模型区暂时不改，仍然展示模型名称和现有模型列表。

  - Hero 不是弱化模型，而是用更有吸引力的问题文案把模型能力提前展示出来。
- 初步问题目录：
  - 如何找出单车变动的罪魁祸首？ -> 使用单车指标变动归因模型。
  - 预算到底偏在哪里？ -> 使用预算实际对比模型。
  - 销量、收入、成本哪个最影响利润？ -> 使用利润敏感性分析模型。

- 可能动作：
  - 在 Hero 右侧或 Hero 下方加入一组轻量问题入口，文案是问题，辅助文字是“使用某某模型”。
  - 入口可以和现有 Hero 主视觉结合，不要变成新的第二屏，也不要挤占 Lucas 主标题。
  - 点击问题入口可以平滑滚动到第二屏财务模型区，或跳转到对应模型，具体再评估。
  - 第二屏财务模型区暂时维持现有模型名称展示，不做问题目录化改造。
- 验证方式：
  - 用户能不能不看长文案，直接知道哪个模型适合哪个问题。
  - Hero 首屏是否比单纯 slogan 更有点击欲望。
  - 第二屏财务模型区是否仍然保持稳定，不因为首屏改造而被打乱。
- 备注：
  - 这条主要影响最上面的 Hero 首屏，不影响首页第二屏财务模型区，也不要求推翻 `/finance` 详情页。
  - 已确认：第二页财务模型区暂时不动，继续保留当前模型名称和现有展示方式。
  - 文案要像真实业务问题，不要像功能说明。

### 3. AI 助手如何成为全站亮点？

- 状态：`保留（模型校对与使用助手）`
- 候选结论：`保留 / 放弃 / 延后 / 需要 Demo`
- 问题：AI 助手要成为全站亮点，但它的亮点应该建立在“帮助用户理解模型、校对口径、选择可视化和组织分析框架”上，而不是泛泛聊天。
- 背景：财务模型可能承载用户自己的经营数据、预算数据或敏感明细。产品内部边界是：AI 默认不分析具体上传数据。但这个边界不需要作为前台卖点反复告诉用户，用户真正需要感知的是 AI 很懂当前模型、能帮他更快上手和校对。
- 阶段性结论：
  - AI 助手保留为全站亮点。
  - AI 可以知道当前页面和当前模型的说明、字段口径、使用步骤、可视化含义。
  - AI 更像“模型使用、口径校对、可视化建议和分析框架助手”，不是数据代分析助手。
  - 前台表达不主动强调“不读取数据、不分析数据”，只在用户问到边界时清楚回答。
- 可能动作：
  - 为每个财务模型配置专属提示词。
  - AI 打开时展示模型相关 starter prompts，例如“这个模型适合解决什么问题？”“我应该上传什么格式？”“这个图表应该怎么看？”。
  - 支持校对类问题：字段口径是否合理、上传表结构是否符合要求、图表选择是否适合表达当前问题。
  - 支持可视化建议：告诉用户某类经营问题适合看瀑布图、趋势图、热力图、结构占比还是敏感性图。
  - 支持分析框架：给用户一套检查顺序，例如先看总差异，再看结构影响，再看单车指标，再看固定扣减。
  - 支持提示词模板：给用户可复制的分析提示词，但不声称已经读取用户数据。
- 验证方式：
  - 在每个模型页面问“这个模型怎么用”，回答必须贴合当前模型。
  - 问“我应该怎么看这个模型的可视化”，AI 能给出对应图表和阅读顺序。
  - AI 能做校对和解释，但不能假装看到了用户上传数据。
  - AI 不应暴露内部索引、fallback、配置等实现语言。
- 待评估风险：
  - 如果未来加入“分析当前数据”，需要明确授权、显著提示、可撤回，并考虑是否只在本地处理摘要。
  - 如果只做固定提示词，回答要避免过度承诺，不能说“我已经看到你的数据”。
- 亮点方向：
  - 上下文感：AI 知道用户在哪个页面、面对哪个模型。
  - 专业感：AI 不是泛聊，而是围绕财务模型、口径、图表和分析框架回答。
  - 可操作感：AI 给出的不是大段解释，而是下一步可以怎么上传、怎么看、怎么校对、怎么汇报。
  - 个人风格：AI 的回答体现 Lucas 的方法论，但语气克制，不冒充资深专家。
- 备注：
  - 这条后续需要重点设计，因为它可能成为全站最有记忆点的功能。
  - 已激活前三条的执行方案另见 `docs/active-home-ai-improvement-plan.md`。
